# 財務授權與個案結案工作坊
### TODO-WORKSHOP-09 + 10 + 11 合併場（全天）

> **這三個是金流與個案生命週期的核心。**  
> 09 定授權層級、10 定核銷憑證格式、11 定個案結案條件。三者順序相依：沒有 09 就不知道誰能核准，沒有 10 就不知道核銷要附什麼，沒有 11 就不知道「結案」代表什麼。

---

| 項目 | 內容 |
|------|------|
| **時間** | 全天，建議 09:00–17:00（7 小時，含午休） |
| **地點** | 有投影設備的室內空間，需要白板 |
| **人數上限** | 12 人 |
| **會前準備截止** | 工作坊前 3 天完成 |

## 參與者

| 角色 | 人數 | 職責 |
|------|------|------|
| 財務幹部／主委 | 2 位 | 定義授權層級、核銷規則 |
| 有結案經驗的志工幹部 | 2–3 位 | 提供個案結案現場經驗 |
| 個案管理師（如有）| 1 位 | 補充個案追蹤視角 |
| 系統設計者 | 1 位 | 即時記錄，確認系統可行性 |
| 記錄者 | 1 位 | 白板拍照、逐字記錄 |

---

## 產出物（工作坊結束時手上要有）

1. **援助金額授權矩陣**——哪個金額層級需要哪個層級主管核准
2. **核銷憑證清單**——各援助類型需要附什麼憑證（發票/收據/照片/簽名）
3. **個案結案條件清單**——AidRecord 全部核銷後的結案檢查項目
4. **爭議處理流程**——案主不同意、憑證遺失、超額申請的處理路徑

---

## 議程

### 09:00　破冰（15 分鐘）
每位幹部說一個讓你印象最深的「難以結案」或「金額爭議」案例。

**主持人引導問題：**
- 最後怎麼解決的？
- 如果有系統，那個情況你希望系統能做什麼？

---

### 09:15　現有授權流程盤點（45 分鐘）
**目標：** 把現在「怎麼核准金額」的隱性規則攤到台面上。

**主持人引導問題：**
- 現在一筆援助金，誰能決定？不同金額門檻不同嗎？
- 需要幾個人同意？有沒有書面紀錄？
- 緊急狀況下（主委不在），誰有代理授權？
- 有沒有「這筆錢我不敢批，需要往上請示」的情況？

**白板工具：** 畫出授權層級金字塔，讓幹部填入各層可核准的金額上限。

```
現場志工     → 最高核准金額：______
地區幹部     → 最高核准金額：______
地區主委     → 最高核准金額：______
總部財務     → 最高核准金額：______
特殊委員會   → 無上限（需 __ 人同意）
```

> 產出：援助金額授權矩陣草稿

---

### 10:00　核銷憑證標準化（45 分鐘）
**目標：** 定義各類援助需要哪些憑證，系統才知道上傳欄位怎麼設計。

**討論各援助類型：**

| 援助類型 | 必要憑證 | 選擇性憑證 | 特殊情況處理 |
|----------|----------|------------|-------------|
| 急難金（現金） | | | |
| 修繕補助 | | | |
| 物資提供 | | | |
| 醫療補助 | | | |
| 長期生活補助 | | | |

**主持人引導問題：**
- 憑證遺失或案主無法提供時怎麼辦？
- 有沒有可以用照片代替的情況？
- 長照案主（老人/失能者）有沒有特殊憑證需求？

> 產出：各援助類型憑證清單

---

### 10:45　休息（15 分鐘）

---

### 11:00　個案結案條件（60 分鐘）
**目標：** 定義「個案結案」到底代表什麼，避免結案後再重開。

**主持人說明：**
> 「系統裡，一個個案（Case）可能跨越多次援助、多個修繕項目、持續追蹤很多年。我們需要定義什麼時候可以『關閉』這個個案。」

**主持人引導問題：**
- 什麼情況下你會說「這個個案結束了」？
- 結案前需要確認哪些事情？（所有援助都核銷了？案主簽字了？修繕完工了？）
- 有沒有「暫時結案」和「永久結案」的差別？
- 結案後如果案主又有需求，是開新案還是重開舊案？
- 有沒有個案應該「長期追蹤」但不算開案的狀態？

**白板工具：** 畫個案生命週期圖

```
[開案] → [評估中] → [援助執行中] → [待結案審核] → [已結案]
                                              ↓
                                        [爭議/暫停]
                                              ↓
                                        [長期追蹤]（不算開案）
```

讓幹部在每個狀態轉換箭頭上填入「誰能觸發轉換」「需要什麼條件」。

> 產出：個案狀態轉換圖 + 每個轉換的條件

---

### 12:00　午休（60 分鐘）

---

### 13:00　爭議與例外情況處理（60 分鐘）
**目標：** 把現實中發生過的爭議案型整理成標準處理流程。

**分組討論（3 組，各 20 分鐘，最後統整）：**

**A 組：金額爭議**
- 案主認為補助金額不夠怎麼辦？
- 超出預算時的處理流程？
- 家族成員有不同意見時？

**B 組：憑證問題**
- 憑證遺失或損毀？
- 廠商不願開發票（現金交易）？
- 修繕品質有爭議，案主不肯簽收？

**C 組：狀態異常**
- 案主失聯怎麼辦？
- 案主過世但家屬繼續申請？
- 同一個案主在不同地區開案？

> 產出：爭議處理流程清單（每種爭議型：觸發條件 + 處理步驟 + 最終決定者）

---

### 14:00　系統流程走讀（45 分鐘）
**目標：** 讓幹部用白板模擬系統操作流程，找出缺漏。

**主持人說明：**
> 「假設今天有一個修繕案，金額 8 萬元，我們走一遍從『個案建立』到『結案』的完整流程，看看每一步系統應該做什麼。」

**走讀步驟：**
1. 現場評估完成 → 建立個案 → 分類損害
2. 建立修繕任務 → 指派承包商 → 里程碑追蹤
3. 完工 → 上傳憑證 → 建立 AidRecord
4. AidRecord 金額 8 萬 → 誰來核准？系統怎麼通知？
5. 核准 → 撥款 → 案主確認收到
6. 所有項目完成 → 結案申請 → 誰審核？
7. 結案確認 → 歸檔

在白板上讓幹部指出「這步驟系統沒有對應欄位」或「這個通知誰會收到」。

> 產出：系統流程缺口清單

---

### 14:45　休息（15 分鐘）

---

### 15:00　報表與查詢需求（45 分鐘）
**目標：** 了解財務幹部需要哪些報表，系統才知道要設計什麼查詢功能。

**主持人引導問題：**
- 每個月你需要看什麼財務報表？
- 上級（主委/總部）問起來，你需要提供哪些數字？
- 有沒有「現在很難算，需要花很多時間整理」的報表？
- 有捐款人或政府審計要求的報表格式嗎？

**設計者記錄報表需求清單：**
```
報表名稱 | 頻率 | 使用者 | 資料範圍 | 格式要求
```

> 產出：報表需求清單（至少 5–8 種）

---

### 15:45　系統試用（45 分鐘）
**目標：** 讓幹部實際操作 AidRecord 建立與核准流程原型。

**觀察重點：**
- 哪個步驟讓幹部感到困惑？
- 有沒有欄位不知道要填什麼？
- 金額輸入介面合理嗎？（幣別、小數點、最大值）
- 核准流程的通知方式（Email/App/SMS）偏好？

> 產出：UI 問題清單 + 通知偏好

---

### 16:30　收尾 + 下一步確認（30 分鐘）

**確認事項：**
1. 四份產出物各由誰整理？截止日期？
2. 授權矩陣需要主委正式批准嗎？走什麼流程？
3. 哪些問題今天沒有答案，需要再諮詢誰？（法務？會計師？）
4. 下次修訂時間？（建議 2 週內）
